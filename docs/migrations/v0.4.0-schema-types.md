# Migrating to v0.4.0: Typed Config Schemas

## What Changed

In v0.4.0, all 88 config fields that previously used `type: "json"` in the editor schema have been converted to proper typed fields (`map`, `array`, or individual sub-fields). This affects three layers:

1. **`wfctl editor-schemas` output** — field types changed in the JSON
2. **Visual editor (workflow-editor)** — form widgets changed from JSON textareas to structured editors
3. **IDE plugins** — hover/completion now provides field-level documentation

## Who Is Affected

| Consumer | Impact | Action Required |
|----------|--------|-----------------|
| **YAML config authors** | None | Your configs are unchanged. The engine parses `map[string]any` regardless of schema type. |
| **Visual editor users** | Improved UX | Fields that were JSON textareas are now structured forms (dropdowns, arrays, key-value editors). No action needed. |
| **`wfctl editor-schemas` consumers** | Field types changed | If your tooling matches on `"type": "json"`, update to handle the new types. See table below. |
| **workflow-editor npm consumers** | `MODULE_TYPES` removed | Switch from `MODULE_TYPES` to `MODULE_TYPE_MAP` (still exported, now sourced from engine schemas). |
| **Plugin authors** | CI enforcement | New plugins cannot ship `FieldTypeJSON` config fields — the CI contract test will fail. Use `FieldTypeMap`, `FieldTypeArray`, or individual typed fields. |

## Runtime Behavior Is Unchanged

The engine reads all module config as `map[string]any` at runtime. The schema types only affect:
- How the visual editor renders config fields
- What `wfctl editor-schemas` reports
- JSON Schema validation via `wfctl schema`

**Your existing YAML configs will continue to work without modification.**

## Schema Field Type Changes

### Conversion patterns applied

| Old Type | New Type | When Used |
|----------|----------|-----------|
| `json` | `map` | Key-value structures (credentials, metadata, labels, VPC configs) |
| `json` | `array` | Lists of objects (routes, records, policies, providers, clients) |
| `json` | Individual sub-fields | Small fixed structures (health checks → `path` + `timeout` + `interval`) |

## New `wfctl editor-schemas` Output Format

The output now includes a `stepSchemas` key:

```json
{
  "moduleSchemas": { ... },
  "stepSchemas": { ... },
  "coercionRules": { ... }
}
```

If your tooling destructures the output, add handling for `stepSchemas` or ignore it:

```go
// Before (still works — stepSchemas is simply ignored)
var output struct {
    ModuleSchemas map[string]*schema.ModuleSchema `json:"moduleSchemas"`
    CoercionRules map[string][]string              `json:"coercionRules"`
}

// After (full access)
var output struct {
    ModuleSchemas map[string]*schema.ModuleSchema `json:"moduleSchemas"`
    StepSchemas   map[string]*schema.StepSchema   `json:"stepSchemas"`
    CoercionRules map[string][]string              `json:"coercionRules"`
}
```

## workflow-editor npm Package Changes

The `MODULE_TYPES` static array (~70 hand-maintained type definitions) has been removed. The `MODULE_TYPE_MAP` export is preserved but now derives from `engine-schemas.json` (279 types) instead of the static array.

```typescript
// Before — still works
import { MODULE_TYPE_MAP } from '@gocodealone/workflow-editor';
const httpServer = MODULE_TYPE_MAP['http.server'];

// Before — REMOVED
import { MODULE_TYPES } from '@gocodealone/workflow-editor';
// MODULE_TYPES no longer exists as a static array

// New alternative
import { getEngineModuleTypes } from '@gocodealone/workflow-editor/utils';
const allTypes = getEngineModuleTypes(); // 279 types from engine-schemas.json
```

## For Plugin Authors

The CI contract test (`schema/schema_contract_test.go`) now enforces zero `FieldTypeJSON` fields across both built-in registries and plugin schema files. If you add a new config field:

```go
// This will FAIL CI:
{Key: "myConfig", Type: FieldTypeJSON, Description: "..."}

// Use a specific type instead:
{Key: "myConfig", Type: FieldTypeMap, Description: "..."}       // for key-value structures
{Key: "myConfig", Type: FieldTypeArray, Description: "..."}     // for lists
{Key: "myConfig", Type: FieldTypeString, Description: "..."}    // for simple values
```

If the field genuinely has no known structure (rare), use `FieldTypeMap` with a descriptive `Description` that documents the expected shape.

## New Features in This Release

### DSL Reference
- `docs/dsl-reference.md` — canonical YAML DSL specification
- `wfctl dsl-reference` — extracts the reference as structured JSON
- Visual editor DSL Reference pane (context-sensitive sidebar)
- IDE hover tooltips and completion descriptions powered by the DSL reference

### Struct-Tag Schema Generator
- `pkg/schema/reflect.go` — `GenerateConfigFields()` produces `[]ConfigFieldDef` from Go struct `editor:"..."` tags
- 5 config structs annotated as initial adoption (`DatabaseConfig`, `HTTPServer`, `RedisNoSQLConfig`, `HealthCheckerConfig`, `MetricsCollectorConfig`)
- Future modules should use `editor` struct tags as the source of truth for config field schemas
