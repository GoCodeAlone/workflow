# Migrating to v0.5.0: Expr Template Syntax

## What Changed

v0.5.0 adds a second expression syntax for pipeline step `config` values. Instead of Go template `{{ }}` expressions, you can now write `${ }` expr expressions. Both syntaxes are supported simultaneously — you can mix them in the same config file or even the same string.

The Go template syntax is **not removed** and continues to work without modification. Migration is optional but recommended for new configs.

## Who Is Affected

| Consumer | Impact | Action Required |
|----------|--------|-----------------|
| **Existing YAML config authors** | None | Your `{{ }}` templates continue to work unchanged. |
| **New config authors** | New option available | Prefer `${ }` syntax for cleaner, more readable configs. |
| **Pipeline step consumers** | No runtime change | The engine evaluates both syntaxes at runtime automatically. |
| **`skip_if` users** | New feature | `skip_if` now accepts expr guards: `skip_if: ${ count == 0 }` |

## Syntax Comparison

| Pattern | Go template (legacy) | Expr syntax (recommended) |
|---------|----------------------|---------------------------|
| Top-level field | `{{ .name }}` | `${ name }` |
| Nested field | `{{ .body.email }}` | `${ body.email }` |
| Step output | `{{ .steps.lookup.id }}` | `${ steps["lookup"]["id"] }` |
| Hyphenated step name | _(not supported cleanly)_ | `${ steps["my-step"]["key"] }` |
| Request header | `{{ .headers.X-Request-Id }}` | `${ trigger["headers"]["X-Request-Id"] }` |
| Query param | `{{ .query.limit }}` | `${ trigger["query"]["limit"] }` |
| Function call | `{{ upper .name }}` | `${ upper(name) }` |
| Equality check | `{{ eq .status "active" }}` | `${ status == "active" }` |
| Boolean AND | `{{ and (eq .x "a") (gt .y 0) }}` | `${ x == "a" && y > 0 }` |
| String concat | _(template join)_ | `${ "Hello " + name }` |
| Conditional skip | _(not available)_ | `skip_if: ${ count == 0 }` |

## Available Namespaces

| Namespace | Contents |
|-----------|----------|
| _(bare name)_ | Top-level context fields (e.g., `${ name }`) |
| `steps` | Outputs from named pipeline steps: `${ steps["step-name"]["key"] }` |
| `trigger` | Trigger data: headers, query params, body — `${ trigger["headers"]["X-Foo"] }` |
| `body` | Alias for `trigger` (convenience) |
| `meta` | Workflow metadata fields |
| `current` | Current step context |

## Automatic Migration

Use `wfctl expr-migrate` to convert an existing config automatically:

```bash
# Preview conversions without writing
wfctl expr-migrate --config app.yaml --dry-run

# Write to a new file
wfctl expr-migrate --config app.yaml --output app-new.yaml

# Rewrite in place
wfctl expr-migrate --config app.yaml --inplace
```

Patterns that cannot be safely auto-converted receive a `# TODO: migrate` comment pointing to the expression that needs manual review.

### What the tool converts automatically

- `{{ .field }}` → `${ field }`
- `{{ .body.name }}` → `${ body.name }`
- `{{ .steps.name.field }}` → `${ steps["name"]["field"] }`
- `{{ eq .x "val" }}` → `${ x == "val" }`
- `{{ ne .x "val" }}` → `${ x != "val" }`
- `{{ gt .x N }}` / `{{ lt .x N }}` → `${ x > N }` / `${ x < N }`
- `{{ index .steps "n" "k" }}` → `${ steps["n"]["k"] }`
- `{{ upper .name }}` → `${ upper(name) }`
- `{{ and (eq ...) ... }}` → `${ ... && ... }`

### What requires manual migration

- Complex nested template pipelines (e.g., `{{ .x | upper | trimSpace }}`)
- Custom template functions not available in the expr engine
- `range`/`with`/`if-else` blocks (these are not valid in step config values; refactor to `skip_if` guards instead)

## Example: Before and After

**Before (Go templates):**
```yaml
pipelines:
  - name: greet
    steps:
      - name: lookup
        type: step.http_call
        config:
          url: "https://api.example.com/users/{{ .user_id }}"
          headers:
            Authorization: "Bearer {{ .token }}"
      - name: respond
        type: step.set
        config:
          message: "Hello, {{ .steps.lookup.name }}!"
```

**After (expr syntax):**
```yaml
pipelines:
  - name: greet
    steps:
      - name: lookup
        type: step.http_call
        config:
          url: "https://api.example.com/users/${ user_id }"
          headers:
            Authorization: "Bearer ${ token }"
      - name: respond
        type: step.set
        config:
          message: "Hello, ${ steps[\"lookup\"][\"name\"] }!"
```

## Using `skip_if` Guards

The `skip_if` field on pipeline steps now accepts expr expressions:

```yaml
steps:
  - name: send-email
    type: step.http_call
    skip_if: ${ notify == false }
    config:
      url: "https://mail.example.com/send"
```

This is new in v0.5.0 and has no Go template equivalent.

## Runtime Behavior

The engine auto-detects which syntax is in use for each value:
- Values containing `${` are evaluated as expr
- Values containing `{{` are evaluated as Go templates
- Values containing both are evaluated by running expr first, then Go template on the result

No configuration changes are needed to enable the new syntax — it is active by default in v0.5.0+.
