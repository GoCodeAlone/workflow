Goal
----

Smoke-load a strict-cutover IaC plugin (i.e. one that uses sdk.ServeIaCPlugin
without ManifestProvider set) against the rebuilt cmd/server binary, exercising
the engine-side disk-manifest fallback path end-to-end. Verify:

  (a) No "validate manifest" error at plugin load time.
  (b) No "proto: unknown field \"_config_dir\"" error during config decode.
  (c) Plugin loads with non-empty Version on the in-engine adapter.

Verdict

LAUNCH gate: PENDING — must run real-subprocess smoke against
workflow-plugin-digitalocean v1.0.11 (or any other strict-cutover IaC plugin
release) before tagging v0.51.3. Recommended pre-release ritual:

    1. Tag workflow main with v0.51.3-rc1.
    2. Build DO plugin against v0.51.3-rc1 SDK.
    3. Stage DO binary + plugin.json at /tmp/wf-plugins-test/digitalocean/.
    4. Run /tmp/wf-server -config example/iac-do-basic-config.yaml \
                          -plugins-dir /tmp/wf-plugins-test
    5. Verify log "plugin \"digitalocean\" loaded successfully" AND no
       "validate manifest" / "unknown field \"_config_dir\"" errors.
    6. If green → promote v0.51.3-rc1 → v0.51.3.
    7. If red → file regression issue, do NOT tag.

Per ADR-0031 §Consequences this gate is a release prerequisite, not a merge
prerequisite, so the PR can land while the launch gate runs against the
to-be-tagged candidate.
