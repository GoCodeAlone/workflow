# Build + Deploy Pipeline Tutorial

This tutorial walks you through building and deploying an application with `wfctl build` and `wfctl ci init`, starting from a single Go binary and expanding to a full polyglot, multi-environment pipeline with SBOM, provenance, and registry retention.

> **Prerequisites**: `wfctl` v0.14.0+, Docker, a DigitalOcean account (or GitHub Container Registry).

---

## 1. Hello World: Go + DO App Platform

Deploy your first Go binary to DigitalOcean App Platform in under 5 minutes.

### Goal

Build a Go binary, push a container image to DOCR, and deploy to DO App Platform staging from a GitHub Actions pipeline generated by `wfctl ci init`.

### Config

```yaml
# infra.yaml
ci:
  build:
    targets:
      - name: server
        type: go
        path: ./cmd/server
    containers:
      - name: api
        method: dockerfile
        dockerfile: Dockerfile
        push_to: [docr]
  registries:
    - name: docr
      type: do
      path: registry.digitalocean.com/myorg
      auth:
        env: DIGITALOCEAN_TOKEN
  deploy:
    environments:
      staging:
        provider: do-app-platform
```

### Commands

```sh
# Validate config
wfctl ci validate --config infra.yaml

# Dry-run build (no docker daemon needed)
WFCTL_BUILD_DRY_RUN=1 wfctl build --config infra.yaml

# Generate GitHub Actions pipeline
wfctl ci init --config infra.yaml --platform github-actions
```

### Expected output

```
[dry-run] docker build --file Dockerfile --tag registry.digitalocean.com/myorg/api:latest .
wrote .github/workflows/ci.yml
wrote .github/workflows/deploy.yml
```

### Common pitfalls

- `DIGITALOCEAN_TOKEN` must be set as a GitHub secret named `DIGITALOCEAN_TOKEN`.
- The Dockerfile must be at the repo root or set `dockerfile: path/to/Dockerfile`.

## 2. Add a UI Target

Extend the pipeline to build a Node.js/Vite frontend alongside the Go backend.

### Goal

Add a `nodejs` target so `wfctl build ui` runs `npm ci && npm run build` before the image step.

### Config addition

```yaml
ci:
  build:
    targets:
      - name: frontend
        type: nodejs
        path: ./ui
        config:
          script: build
          node_version: "20"
```

### Commands

```sh
WFCTL_BUILD_DRY_RUN=1 wfctl build ui --config infra.yaml
WFCTL_BUILD_DRY_RUN=1 wfctl build --config infra.yaml   # chains go → ui → image → push
```

### Expected output

```
[dry-run] npm ci && npm run build (cwd: ./ui)
built frontend → [ui/dist]
```

### Common pitfalls

- `package-lock.json` must exist; otherwise `npm ci` is replaced with `npm install` and a security lint warning is emitted.

## 3. Multi-environment: Staging + Prod

Add a production environment with approval gate so deploys require manual sign-off.

### Goal

Two environments: `staging` (auto-deploy) and `prod` (manual approval via GitHub environment protection).

### Config addition

```yaml
ci:
  deploy:
    environments:
      staging:
        provider: do-app-platform
      prod:
        provider: do-app-platform
        requireApproval: true
```

### Generated deploy.yml shape

```yaml
deploy-staging:
  needs: [build-image]
  ...
deploy-prod:
  needs: [deploy-staging]
  environment: prod          # triggers GitHub environment protection
  ...
```

### Common pitfalls

- The `prod` GitHub environment must exist with a required reviewer configured.
- `wfctl ci init` chains envs alphabetically; non-approval envs deploy first.

## 4. Polyglot: Go + Rust + Python

Add Rust and Python targets. Built-in builders handle Go and Node.js; Rust and Python require external builder plugins.

### Goal

Build a Go API, a Rust CLI, and a Python data-processing worker in one pipeline.

### Config

```yaml
ci:
  build:
    targets:
      - name: server
        type: go
        path: ./cmd/server
      - name: cli
        type: custom
        config:
          command: "cargo build --release --manifest-path cli/Cargo.toml"
          outputs: [cli/target/release/cli]
      - name: worker
        type: custom
        config:
          command: "pip install -r worker/requirements.txt && python worker/build.py"
          outputs: [worker/dist/]
```

### Common pitfalls

- Use `custom` type for Rust/Python until `workflow-plugin-builder-rust` / `workflow-plugin-builder-python` are installed.
- `SecurityLint` on `custom` type always emits a `warn` finding — this is expected.

## 5. CGO with Embedded SQLite

Build a Go binary with CGO enabled and a statically linked SQLite.

### Goal

Produce a single-binary server with embedded SQLite using a controlled builder image.

### Config

```yaml
ci:
  build:
    targets:
      - name: server
        type: go
        path: ./cmd/server
        config:
          cgo: true
          link_mode: external
          builder_image: golang:alpine
          cgo_cflags: "-D_LARGEFILE64_SOURCE"
          extra_flags: ["-trimpath"]
```

### Common pitfalls

- Omitting `link_mode` when `cgo: true` triggers a `SecurityLint` warn.
- `builder_image` must be a known-safe image (`golang:alpine`, `golang:bookworm`).

## 6. Custom Dockerfile

Override the default `dockerfile: Dockerfile` with a multi-stage build and BuildKit secrets.

### Goal

Use a custom Dockerfile path, pass a private Go module token via BuildKit secret, and push to DOCR.

### Config

```yaml
ci:
  build:
    containers:
      - name: api
        method: dockerfile
        dockerfile: docker/api.Dockerfile
        secrets:
          - id: goprivate_token
            env: GOPRIVATE_TOKEN
        build_args:
          BASE_IMAGE: golang:1.26-alpine
        platforms: [linux/amd64, linux/arm64]
        push_to: [docr]
```

### Commands

```sh
WFCTL_BUILD_DRY_RUN=1 wfctl build image --config infra.yaml
# [dry-run] docker build --file docker/api.Dockerfile --platform linux/amd64,linux/arm64
#   --secret id=goprivate_token,env=GOPRIVATE_TOKEN --build-arg BASE_IMAGE=golang:1.26-alpine .
```

### Common pitfalls

- BuildKit must be enabled: `DOCKER_BUILDKIT=1` is set automatically by `wfctl build image`.
- Multi-arch builds require `docker buildx` with a multi-platform builder.

## 7. External Image

Reference a pre-built base image from an upstream registry without rebuilding it.

### Goal

Pull a versioned base image from DOCR via a `tag_from` env chain instead of building it locally.

### Config

```yaml
ci:
  build:
    containers:
      - name: base
        external: true
        push_to: [docr]
        source:
          ref: registry.digitalocean.com/myorg/base
          tag_from:
            - env: BASE_IMAGE_TAG
            - command: "git describe --tags --abbrev=0"
```

### Commands

```sh
BASE_IMAGE_TAG=v1.2.3 WFCTL_BUILD_DRY_RUN=1 wfctl build image --config infra.yaml
# [dry-run] external image: base → registry.digitalocean.com/myorg/base:v1.2.3
```

### Common pitfalls

- `external: true` skips `docker build` entirely — it only resolves and records the image ref.
- Tag resolution order: env var wins over shell command.

## 8. Multi-registry: DOCR + GHCR

Push the same image to both DigitalOcean Container Registry and GitHub Container Registry.

### Goal

Single build, dual push — useful for open-source images on GHCR alongside private DOCR.

### Config

```yaml
ci:
  registries:
    - name: docr
      type: do
      path: registry.digitalocean.com/myorg
      auth:
        env: DIGITALOCEAN_TOKEN
    - name: ghcr
      type: github
      path: ghcr.io/myorg
      auth:
        env: GITHUB_TOKEN
  build:
    containers:
      - name: api
        method: dockerfile
        dockerfile: Dockerfile
        push_to: [docr, ghcr]
```

### Commands

```sh
wfctl registry login --config infra.yaml   # logs into both registries
WFCTL_BUILD_DRY_RUN=1 wfctl build --config infra.yaml
# [dry-run] docker push registry.digitalocean.com/myorg/api:latest
# [dry-run] docker push ghcr.io/myorg/api:latest
```

## 9. Retention Policy

Automatically prune old images on a weekly schedule to control registry costs.

### Goal

Keep only the 20 most recent image tags in DOCR; delete untagged manifests on a cron.

### Config

```yaml
ci:
  registries:
    - name: docr
      type: do
      path: registry.digitalocean.com/myorg
      auth:
        env: DIGITALOCEAN_TOKEN
      retention:
        keep_latest: 20
        untagged_ttl: 168h
        schedule: "0 7 * * 0"   # Sundays at 07:00 UTC
```

### Generated workflow

`wfctl ci init` emits `.github/workflows/registry-retention.yml`:

```yaml
on:
  schedule:
    - cron: '0 7 * * 0'
jobs:
  prune:
    steps:
      - run: wfctl registry prune --registry docr
```

### Dry-run

```sh
WFCTL_BUILD_DRY_RUN=1 wfctl registry prune --registry docr --config infra.yaml
# [dry-run] doctl registry garbage-collection start --force --include-untagged-manifests
# [dry-run] doctl registry repository list-tags (keep latest 20, preserve 'latest')
```

## 10. Local Dev with wfctl dev up

Run `wfctl dev up` for fast local iteration with relaxed security defaults and Docker layer caching.

### Goal

`wfctl dev up` applies `environments.local.build` overrides: no hardening, no SBOM, local Docker cache.

### Config

```yaml
environments:
  local:
    build:
      targets:
        - name: server
          type: go
          path: ./cmd/server
          config:
            race: true
            ldflags: ""
```

### Commands

```sh
WFCTL_BUILD_DRY_RUN=1 wfctl dev up
# resolved: security.hardened=false, security.sbom=false (fast iteration)
# [dry-run] go build -race ./cmd/server
```

### Common pitfalls

- `environments.local.build.security` can be set explicitly to override the fast-iteration defaults.
- Container targets get `cache.from=[{type:local}]` injected automatically.

## 11. Custom Health Checks

Configure gRPC, TCP, and composite health checks for your deployment environments.

### Goal

Ensure DO App Platform waits for the gRPC health endpoint before marking a deploy successful.

### Config

```yaml
ci:
  deploy:
    environments:
      staging:
        provider: do-app-platform
        healthCheck:
          path: /healthz
          timeout: 30s
```

> For gRPC and TCP health checks, use the DO App Platform console or a `preDeploy` hook that polls `grpc_health_probe`.

### Common pitfalls

- HTTP health checks only; gRPC requires a sidecar or pre-deploy hook.
- `timeout` must parse as a Go duration string (`30s`, `2m`).

## 12. Deploy Hooks

Run database migrations or cache warmup before traffic is switched to the new version.

### Goal

Run `wfctl ci run --phase migrate` before each staging deploy.

### Config

```yaml
ci:
  deploy:
    environments:
      staging:
        provider: do-app-platform
        preDeploy:
          - wfctl ci run --phase migrate --env staging
```

> `preDeploy` commands are injected as `run:` steps before the `wfctl ci run --phase deploy` step in generated workflows.

### Common pitfalls

- Migration failures abort the deploy; ensure migrations are idempotent.

## 13. Non-software Builds: Hugo Static Site

Use the `custom` builder type to build a Hugo static site and push its Docker image.

### Goal

Generate a static site with Hugo, containerize it with Nginx, and deploy.

### Config

```yaml
ci:
  build:
    targets:
      - name: site
        type: custom
        config:
          command: "hugo --minify --destination ./public"
          outputs: [./public/]
          timeout: 120s
    containers:
      - name: site-server
        method: dockerfile
        dockerfile: Dockerfile.nginx
        push_to: [docr]
```

### Common pitfalls

- Hugo must be installed in the CI runner. Add `apt-get install hugo` to a pre-step or use the `GoCodeAlone/setup-hugo` action.

## 14. Alternative Deploy: Kubernetes

Deploy to a local minikube cluster with `wfctl dev up --k8s` or to production GKE/EKS via a custom provider plugin.

> Kubernetes deploy targets are out-of-scope for v0.14.0 built-ins. See the `workflow-plugin-deploy-k8s` plugin in the registry.

### Workaround for v0.14.0

```yaml
ci:
  deploy:
    environments:
      prod:
        provider: custom
        preDeploy:
          - kubectl set image deployment/api api=$IMAGE_REF
```

## 15. Signing + Attestation

Enable Sigstore cosign signing and SLSA-3 provenance attestation on every image.

### Goal

Every image built by `wfctl build` carries a SBOM, provenance, and cosign signature.

### Config

```yaml
ci:
  build:
    security:
      hardened: true
      sbom: true
      provenance: slsa-3
      sign: true
```

> These are the **default** values — no explicit config is needed. To disable for local dev, use `environments.local.build.security.hardened: false`.

### What wfctl does

1. Runs `syft <image-ref> -o cyclonedx-json > sbom.json` after each image build.
2. Attaches SBOM via `oras attach` (or `cosign attach sbom` if oras is unavailable).
3. Passes `--attest=type=provenance,mode=max` to `docker buildx build` (T33).

### Common pitfalls

- `syft` and `oras`/`cosign` must be on PATH in the CI runner.

## 16. Debugging a Failed Deploy

Diagnose and recover from common `wfctl build` and `wfctl ci run` failures.

### Symptoms and fixes

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `doctl registry login: exit status 1` | `DIGITALOCEAN_TOKEN` not set | Ensure secret is configured in GH Actions |
| `push_to references undeclared registry` | Registry name typo | Match `push_to:` entry to `registries[].name` |
| `go builder: path is required` | Target missing `path:` field | Add `path: ./cmd/server` |
| `custom builder cannot enforce hardening` | Using `type: custom` | Expected warning; not an error |
| Security audit exits 1 | Critical finding (e.g. `USER root`) | Fix Dockerfile; see Section 7 of manual |

### Useful commands

```sh
# Validate config before building
wfctl ci validate --config infra.yaml

# Dry-run to preview all planned steps
WFCTL_BUILD_DRY_RUN=1 wfctl build --config infra.yaml

# Security audit
wfctl build --security-audit --config infra.yaml
```
