# Plugin Contract And Registry Design

Date: 2026-04-25

## Goal

Make plugins installable, verifiable, discoverable, and compatible through one canonical manifest contract shared by local plugin repos, `workflow-registry`, `workflow-cloud-registry`, and `wfctl`.

## Problem

Plugin repos currently use several incompatible manifest shapes:

- nested `capabilities` object, which matches the registry direction
- top-level `moduleTypes` and `stepTypes`
- array-style `capabilities`
- provider-specific `resources`
- missing `plugin.json`
- inconsistent names, such as short names (`admin`, `slack`, `azure`) mixed with full repo names (`workflow-plugin-auth`)
- placeholder template identity (`workflow-plugin-TEMPLATE`)

Most plugins have release scaffolding, but compatibility is still fragile. The broad matrix found failures from checksum mismatch, missing `go.sum`, and API-signature drift. Those are release-readiness issues, not just local test issues.

## Canonical Manifest

The canonical `plugin.json` shape should match the registry schema and be accepted by `wfctl` without transformation:

```json
{
  "name": "workflow-plugin-example",
  "version": "0.1.0",
  "author": "GoCodeAlone",
  "description": "Example plugin",
  "type": "external",
  "tier": "community",
  "license": "MIT",
  "minEngineVersion": "0.19.0",
  "repository": "https://github.com/GoCodeAlone/workflow-plugin-example",
  "capabilities": {
    "configProvider": false,
    "moduleTypes": [],
    "stepTypes": [],
    "triggerTypes": [],
    "iacProvider": {
      "name": "example",
      "resourceTypes": []
    },
    "cliCommands": [],
    "buildHooks": []
  }
}
```

Compatibility readers can support old shapes for one transition window, but writers and templates must emit the canonical shape only.

## Naming

Use one display-safe and filesystem-safe plugin identity:

- repo: `workflow-plugin-auth`
- manifest `name`: `workflow-plugin-auth`
- install directory: canonical normalized name, with documented alias support if short names are retained
- registry entry key: canonical name

Short aliases (`auth`, `slack`, `teams`) can exist as registry metadata, but should not replace the canonical name.

## Version And Compatibility

Each plugin release should declare:

- `version`
- `minEngineVersion`
- optional `maxEngineVersion` only when there is a known incompatibility
- exact workflow dependency in `go.mod`
- compatibility test against the current supported workflow release

The ecosystem should publish a compatibility matrix:

| Plugin | Manifest Valid | Tests | Workflow Dep | Min Engine | Release Artifacts |
|---|---|---|---|---|---|

This should be generated by `wfctl audit plugins` in a follow-up.

## Integrity

Install integrity should be fail-closed:

- `.wfctl-lock.yaml` records exact version, source, platform artifact URLs, and SHA-256.
- GitHub release assets with checksums are verified automatically.
- Plugins with `verify` policy emit `install_verify` hooks before extraction.
- A checksum mismatch is a release blocker.

The `workflow-plugin-agent` checksum mismatch found during audit is a high-priority blocker because Go module authentication is explicitly warning that downloaded bytes differ from `go.sum`.

## Template Quality

`workflow-plugin-template` and `workflow-plugin-template-private` are part of the product contract because new plugins inherit their shape. They must:

- build and test cleanly from a fresh checkout
- contain no placeholder module path in `go.mod`
- emit canonical manifest shape
- include `go.sum`
- include a sample conformance test

## Testing

Immediate audit command:

```sh
for d in /Users/jon/workspace/workflow-plugin-*; do
  [ -f "$d/go.mod" ] || continue
  (cd "$d" && GOWORK=off go test ./...)
done
```

Expected near-term result after remediation: every public, non-template plugin passes; template repos pass as generated scaffolds; private game plugins may be reported separately but should not be hidden.

Follow-up commands:

```sh
wfctl plugin validate --all --repo-root /Users/jon/workspace
wfctl audit plugins --repo-root /Users/jon/workspace --json
wfctl audit plugins --registry /Users/jon/workspace/workflow-registry
```

## Acceptance Criteria

- All plugin manifests validate against one schema.
- Legacy manifest shapes are read-only compatibility paths.
- New plugin scaffolds produce canonical manifests and clean tests.
- Release-readiness audit reports manifest validity, test status, workflow dependency, and checksum status.
- Registry manifests and local plugin manifests do not drift silently.
