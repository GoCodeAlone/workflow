- **[Missing failure mode]** §T7.14 dedup helper line 1758 runs `gh issue list --label "$LABEL" --state open --json number --jq '.[0].number // empty'` then comments on the first match. **There is no "must be created by the helper" guard.** Two failure modes: (1) a maintainer reopens a closed `conformance-leak-incident` issue intentionally for tracking history; the helper appends comments to that historical issue, polluting the audit trail. - **[Missing failure mode]** §T1.5 unconditional postcondition `defer func() { ... }()` runs even if the apply panics partway through. If `currentInputSnapshot(...)` itself reads env state that the apply has unset (e.g., a sub-action that calls `os.Unsetenv("STAGING_PG_PASSWORD")` for credential-cleanup reasons), the postcondition will fingerprint an empty value and report a false-positive "drift" between plan-time fingerprint and apply-time empty. The plan offers no test for this case. Worse: if `currentInputSnapshot` itself panics (env-provider closure captured a freed resource), the deferred panic compounds the original error and corrupts the apply result. **Recommendation:** wrap the deferred postcondition body in `recover()`; on panic, set `result.InputDriftReport = nil` + log a warning. Add a TDD test `TestApply_Postcondition_PanicDoesNotCorruptResult` exercising a panicking env-provider stub.
