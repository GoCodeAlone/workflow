- **[Unstated assumptions / Missing failure modes] Â§3 Replace action â€” gRPC boundary cost not accounted for.** The new ComputePlan now calls `provider.ResourceDriver(spec.Type).Diff(ctx, spec, currentOut)` once per existing resource. **Options the author may not have considered:**

1. **Ship V1/V2 dual-interface (revisit Approach 2) with codemods on the V2 side.** Trade-off: short-term complexity (two interfaces, adapter shims) but each plugin migration is a per-plugin PR that can land independently, ride its own CI, and roll back without affecting others. The author rejected this as "long tail of V1 edge-case divergence" â€” but the long tail is exactly what conformance suite W-7 catches. With conformance enforcing semantic equivalence, the V1 wrapping cost is bounded. Revisit this if user-intent drift (Important #1) confirms the user wanted Approach 2.

2. **Cache Diff results during ComputePlan via a content-addressed key.** Trade-off: adds local cache (`~/.cache/wfctl/diff/`) keyed by `(plugin-version, resource-type, sha256(spec.Config), sha256(current.Outputs))`. For repeat plan runs (operator iterates `wfctl infra plan` while authoring config), 50-resource plans go from 50 gRPC roundtrips to 0. Catches the gRPC cost (Critical #1) without changing the plugin contract. Cost: cache invalidation on plugin upgrade (manageable via plugin-version key).

3. **Move `Provider.ValidatePlan` to the `iaclint`/conformance package as a generic constraint-DSL evaluator with per-provider rules registered via plugin manifest.** Trade-off: pushes constraint-validation into a declarative DSL (`{type: "vpc-region-match", left: "$.app.region", right: "$.vpc.region"}`); providers register rules without writing Go. Author rejected this as "premature abstraction" but with 4 active providers the DSL pays back fast. Counter-argument: the Go hook is genuinely simpler to ship in W-4. Revisit at the conformance-suite extraction milestone (W-7).
