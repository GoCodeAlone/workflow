# Phase 2.3 — ActionStatus compensation enums + mapping fixes

**Date:** 2026-05-17
**Status:** Draft (autonomous brainstorming output)
**Related:** workflow#698, workflow#640 (Phase 2 PR #694), workflow#695 (Phase 2.5 PR #697)
**ADRs binding:** decisions/0024-iac-typed-force-cutover.md (no compat shim), decisions/0040 (Phase 1 invariants)

## Goal

Define the 3 ActionStatus enum values that Phase 2 reserved (tags 4 + 5) and Phase 2.5 deferred (tag 6 SKIPPED candidate). Fix the engine's mapping in `iac/wfctlhelpers/apply.go` so per-action exit paths use the correct status (SKIPPED for not-attempted; COMPENSATION_FAILED for post-hook-failure-after-driver-success).

## Scope (minimal — workflow-only, single PR)

**1 PR / workflow v0.56.0 (minor — adds enum values + observable status mapping changes for previously-wrong paths).**

Phase 2.3 does NOT introduce engine-side auto-compensation logic. ACTION_STATUS_COMPENSATED is defined and reserved for FUTURE work (plugins/engine that implement actual rollback). For now, post-hook-failure paths map to COMPENSATION_FAILED ("cloud-side work happened; hook failed; nobody compensated; state preservation required") — actionable for operators.

Plugin cascade NOT needed: plugins don't emit these statuses today; engine populates them server-side. Workflow ships the enum + mapping; plugins adopt at their own pace if/when they implement own compensation logic.

## Decisions

### A. Enum values (define all 3)

```protobuf
enum ActionStatus {
  ACTION_STATUS_UNSPECIFIED = 0;
  ACTION_STATUS_SUCCESS = 1;
  ACTION_STATUS_ERROR = 2;
  ACTION_STATUS_DELETE_FAILED = 3;
  ACTION_STATUS_COMPENSATED = 4;          // NEW Phase 2.3 (reserved-for-future-emission; not emitted by workflow v0.56.0)
  ACTION_STATUS_COMPENSATION_FAILED = 5;  // NEW Phase 2.3 (replaces post-hook-fail Error/DeleteFailed)
  ACTION_STATUS_SKIPPED = 6;              // NEW Phase 2.3 (replaces pre-dispatch ctx-cancel/JIT-fail/driver-resolve-fail Error/DeleteFailed)
}
```

COMPENSATED is defined for forward-compat clarity but NOT EMITTED by workflow v0.56.0. Reserved for plugins that implement own compensation logic OR a future workflow phase that adds engine-side auto-compensation.

### B. Engine mapping fix (apply.go)

Refactor `mapDispatchErrToStatus` from single-function to phase-aware. Replace single helper with 3 phase-specific helpers:

```go
// statusForPreDispatchSkip returns SKIPPED — action was never attempted at
// the driver. Used for ctx-cancel, JIT-substitution-fail, driver-resolve-fail
// paths. Cloud-side state unchanged from pre-apply.
func statusForPreDispatchSkip() interfaces.ActionStatus {
    return interfaces.ActionStatusSkipped
}

// statusForDispatchError returns Error (non-delete) or DeleteFailed (delete).
// Used when driver's Create/Update/Delete RPC returned an error. The action
// was attempted; cloud-side state may be partially mutated. For delete
// actions, DELETE_FAILED instructs wfctl to preserve state.
func statusForDispatchError(actionType string) interfaces.ActionStatus {
    if actionType == "delete" {
        return interfaces.ActionStatusDeleteFailed
    }
    return interfaces.ActionStatusError
}

// statusForPostHookFailure returns COMPENSATION_FAILED — driver succeeded
// but wfctl-side hook (OnResourceApplied / OnResourceDeleted) failed.
// Cloud-side work IS done; operator must verify state; may need manual
// compensation. State preservation required regardless of action type.
func statusForPostHookFailure() interfaces.ActionStatus {
    return interfaces.ActionStatusCompensationFailed
}
```

Call sites in apply.go (exact line numbers verified at plan-write per ADR 0035):
- ctx-cancel inside loop → `statusForPreDispatchSkip()`
- JIT-substitution-fail → `statusForPreDispatchSkip()`
- driver-resolve-fail → `statusForPreDispatchSkip()`
- dispatch error from driver → `statusForDispatchError(action.Action)`
- post-delete-hook-fail → `statusForPostHookFailure()`
- post-apply-hook-fail → `statusForPostHookFailure()`

The existing `mapDispatchErrToStatus` function deleted (replaced by 3 phase-specific helpers).

### F. ADR 0024 compat impact

The mapping refactor is observable to wfctl callers downstream of `ApplyResult.Actions`:
- Previously, `ActionStatusError` could mean "driver failed" OR "post-hook failed". Now ONLY means "driver failed".
- Previously, `ActionStatusDeleteFailed` could mean "delete failed" OR "post-delete-hook failed" OR "ctx-cancel of a delete". Now ONLY means "delete dispatch failed".

Per ADR 0024 hard-cutover: intentional semantic correction; the previously-wrong mapping was a bug. Consumers update to check `{Error, DeleteFailed, CompensationFailed, Skipped}` for "this action did not produce desired end-state". Document in `interfaces/iac_state.go` ActionStatus godoc.

## Considered alternatives

### Option 1: Add engine auto-compensation (rejected)

Engine on post-hook-failure calls inverse RPC (Delete on just-created; Create on just-deleted). Records COMPENSATED or COMPENSATION_FAILED.

Rejected:
1. Re-entrant Delete mid-flight has known partial-state hazards
2. Some resources not compensable (DBs with data; uploaded files)
3. Needs real-world driver to inform design (zero today)
4. Per-resource-type opt-in adds plugin-side complexity

Defer to future phase with demonstrated need.

### Option 2: Skip mapping refactor; only add enum values (rejected)

Add enum values but don't change apply.go. Tags sit dormant.

Rejected: defeats the point. Cycle-1/2/3 findings were ABOUT misclassification. ADR 0040 invariant 2 (failed-delete preservation) violated when ctx-cancel maps to DeleteFailed for never-attempted action.

### Option 3: PICKED — minimal mapping + enum addition

Pure workflow change. Single PR. Single tag bump. No plugin cascade. Defers auto-compensation to future phase.

## Assumptions (load-bearing)

1. **No wfctl-side consumer keys behavior off `ActionStatusError` alone.** Verified at plan-write via grep across cmd/wfctl/ + interfaces/.
2. **`ActionStatusDeleteFailed` consumers expect "delete failed AND state-preservation-required"**. Re-mapping ctx-cancel-of-delete from DeleteFailed to SKIPPED is correct because the resource was never touched.
3. **Plugin emission of new enum values is opt-in**. Older plugins (aws v1.2.1 + azure v1.2.1 + gcp v1.2.0 + DO v1.4.0) continue emitting Success/Error/DeleteFailed unchanged.
4. **No engine-side auto-compensation in Phase 2.3**. COMPENSATED is forward-compat reserved.
5. **`mapPBActionStatusToInterface` fail-closed default protects old wfctl against new plugin enum values**. Phase 2 cycle-1 C-3 fix preserved.
6. **`iac_typed_adapter_test.go` table covers UNSPECIFIED-fails-closed**. Already exists from Phase 2.
7. **`interfaces.ActionStatus` Go iota values match proto tag numbers**. Verified Phase 2; preserved here.

## Self-challenge findings

1. **Laziest plausible?** Skip Phase 2.3 entirely; document in MEMORY.md as known-issues. Cost: keeps misclassification bug. Rejected.
2. **Most fragile assumption?** #1 (no consumer keys off Error alone). Mitigated by grep verification.
3. **YAGNI sweep?** Considered + rejected: per-resource compensation policy; driver-level Compensable interface; engine-side auto-compensation; pluggable strategies. All deferred per "demonstrated need" discipline.
4. **What fails first under partial failure?** Plugin emitting COMPENSATED in v_next against workflow v0.55.0 — wfctl fail-closed rejects. INTENDED per ADR 0024.
5. **Repo precedent conflict?** Follows Phase 2 + 2.5 enum-extension precedent. None.
